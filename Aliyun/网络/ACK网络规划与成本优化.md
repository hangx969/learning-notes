# ACK 网络规划与成本优化

> 来源：[踩过网段坑才懂：K8s 网络规划与成本优化的底层逻辑](https://mp.weixin.qq.com/s/caUX_RVVPSs1fwq4UEOpog)

核心观点：**账单上的大头往往藏在网络规划里**。IP 规划不合理导致集群重建、跨 AZ 流量悄悄吃掉 20%+ 预算、公网 SLB 滥用导致费用翻倍——这些都是网络层的隐性成本。

---

## 网段规划为什么和钱有关

| 规划失误 | 后果 | 成本影响 |
|----------|------|----------|
| Pod CIDR 过小 | 扩容顶不住，被迫新建集群迁移 | 新集群资源 + 运维时间 + 业务中断风险 |
| CIDR 重叠 | 多集群互通只能绕路（走公网或跨 AZ） | 跨区域传输费激增 |
| Pod 网段过大 | 路由表暴涨 | CPU/内存额外消耗 |

## 网段设计最佳实践

1. **分区分段，隔离职责**：数据库、ECS、SLB、K8s 节点、Pod 各用独立网段，方便排查和成本统计
2. **Pod 网段尽量大，Service 网段相对小**：Pod 数量远大于 Service，合理分配可减少扩容时的被动成本
3. **Service CIDR 与 VPC CIDR 不重叠**：重叠后网络冲突，修复成本极高
4. **多集群提前留足空间**：每个集群使用不同网段，方便未来通过 VPC Peering 或 CEN 打通

> [!warning] 踩坑提示
> Service CIDR 与 VPC CIDR 重叠是常见坑点，一旦发生只能重建集群，无法在线修复。

## 流量成本优化

ACK 中网络费用三大来源：**跨 AZ 流量费**、**公网出口费**、**SLB 转发费**。

| 策略 | 做法 | 原理 |
|------|------|------|
| 单 AZ 优先 | 能不跨 AZ 就别跨，高可用通过节点分布实现 | 跨 AZ 流量按量计费 |
| 公网流量要控 | 内部服务走内网，公网暴露仅针对必须的服务 | 避免"一股脑上公网 SLB" |
| SLB 精打细算 | 内外网 SLB 分开用；流量大的内部服务用 NodePort + 内部调度 | 减少 SLB 转发费 |

## 网络插件选型：Flannel vs Terway

| 维度 | Flannel | Terway |
|------|---------|--------|
| 复杂度 | 简单轻量 | 基于 VPC ENI，弹性更强 |
| 扩展性 | 有限 | 能避免 IP 不足问题 |
| 适用规模 | 小规模集群，省心省钱 | 中大型集群，前期多花后期省大钱 |
| 成本逻辑 | 短期低成本 | 减少后期迁移和 IP 紧张的风险 |
