## 💰 Kubernetes 成本规划实战（二）：日志与资源配额

在上一篇《节点与 Ingress》中，我们聊了 ACK 集群里最容易烧钱的两个地方。  
这次我们聚焦两个同样关键、但经常被忽略的成本大户：

- **日志**
- **资源配额（requests/limits）**

很多团队不重视这两点，结果就是：  
👉 存储费用一路飙升，Pod 利用率严重失衡。

## 🗂 日志：省钱的第一刀

在阿里云 ACK 上，日志相关费用主要来自：

- **存储（PVC + 云盘）**
- **日志收集与转发（Log Service / 采集 Agent 占资源）**

### 我们的实践

1. **日志写入本地磁盘**
- Pod 日志挂载到 Node 的本地磁盘，而不是云盘
	- 本地磁盘 I/O 更便宜，释放 Pod 时自动清理
3. **定时清理日志**
- 用 `cronjob` 或 Node 脚本定时清理本地日志
	- 默认只保留 **7 天** ，够排查问题了
5. **集中日志节点**
- 如果日志量很大，就单独准备几个 Node 节点
	- 打上标签，把日志收集任务调度过去，避免影响业务节点
7. **和开发协同日志规范**
- ✅ 错误日志、关键链路日志必须收集
	- ❌ Debug、大量无用的 Info 日志不采集
- 不是什么日志都收集：
	- 提前约定日志大小和格式，避免出现“几百 GB 的 Debug 日志”

**优化结果** ：  
日志费用下降 **40%+** ，存储压力大幅缓解。

## ⚙️ 资源配额（requests/limits）：利用率翻倍的关键

另一个大坑是 requests/limits 配置。  
很多开发团队凭感觉写配置：要么太保守，导致 Pod 被频繁驱逐；要么太豪横，结果节点利用率只有 30%。

我们做法是：  
👉 基于 **历史监控数据（P95 指标）** 来配置 requests/limits，避免浪费。

### 推荐配置示例

#### 🔹 Golang 服务

并发量低：

```
1limits:2  cpu: 500m3  memory: 512Mi4requests:5  cpu: 100m6  memory: 128Mi
```

并发量高：

```
1limits:2  cpu: 2000m3  memory: 4096Mi4requests:5  cpu: 100m6  memory: 128Mi
```

#### 🔹 PHP 项目

```
1limits:2  memory:"512Mi"3  cpu:"500m"4requests:5  memory:"256Mi"6  cpu:"100m"
```

#### 🔹 Java 服务（重点）

Java 是最常见的资源黑洞：  
• 堆内存配置不合理 → OOM  
• requests 配置过小 → GC 频繁，响应慢  
• limits 配置过高 → 资源长期闲置

建议：  
1\. 根据 JVM 参数（如 -Xmx, -Xms）来设置 Pod 内存  
2\. 一般来说：requests.memory ≈ Xms，limits.memory ≈ Xmx + 20%  
3\. CPU requests 不要设太小，否则容易被压缩

📉 成本优化效果

经过日志和资源配额优化，我们效果非常明显：  
• 日志存储费用下降 40%+  
• 节点利用率从 30–40% 提升到 70%+  
• Pod OOM 和频繁驱逐情况明显减少  
• 整体云账单又省下了一波

## 🎯 总结

```
•    日志：
•    挂本地、定时清理、分节点、收集规范化
•    资源配额：
•    基于监控数据配置，不靠感觉
•    Java 特别注意堆内存和 requests/limits 匹配
```

一句话总结：  
👉 日志 + 配额优化，是 Kubernetes 成本控制的第二个关键点
