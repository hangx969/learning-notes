以下是完整整理。

## 一、每次打开新会话，Claude 都从零开始

Mayank 开篇没有讲功能，讲的是一个每个人都在经历、但很少有人意识到的隐性成本：

> "事实是这样的：Claude 不会记住你。每次你打开一个新会话，内存清零。"

它不知道你的名字，不知道你是干什么的，不知道你的写作风格，不知道你上周花了三个小时反复纠正过的那些问题。所以你得花几分钟从头解释——或者你不解释，Claude 给你的东西跟你实际的工作方式完全对不上。

这种成本每天都在发生：重复自己、纠正同样的错误、第一百遍解释自己的偏好。"不是因为 Claude 不好，是因为它没有你的说明书。"

CLAUDE.md 就是这份说明书。它在你的项目文件夹里，Claude 会在每个会话开始时自动读取。设置一次，永久生效。

---

## 二、CLAUDE.md 不只是开发者的工具

Mayank 在进入指令清单前先打破了一个最常见的误区：

> "这不仅仅是开发者的工具。"

- **写作者** 用它锁定文风，确保 Claude 写出来的东西不会跑调
- **营销人** 用它定义目标受众，避免 Claude 写出千篇一律的套话
- **研究人员** 用它设定信息呈现结构
- **创业者** 用它给 Claude 提供完整公司背景，确保所有输出符合实际情况

每一个认真使用 Claude 的人，都应该在第一次正式使用之前设置这个文件。

**创建只需两步** ：在项目文件夹新建文件，命名为 `CLAUDE.md` （全大写，无空格）。用任意文本编辑器打开，先写 3-4 条解决最频繁痛点的指令。保存。从下一条消息开始，Claude 自动读取。无需插件，无需配置。

## 三、Part 1：修复 Claude 的沟通方式（指令 1-4）

### 指令 1：彻底禁掉废话开场白

Claude 的默认回复总喜欢以"好问题！""当然！""绝对！""我很乐意帮你！"开头。零信息量的短语每天都在浪费你的时间。

> "回答问题时，千万不要用'好问题！''当然！''绝对！'之类的客套话开头。直接给出答案，不寒暄、不铺垫、即刻提供所需信息。"

一条指令，永久清零。

### 指令 2：做事之前先给选项

Claude 的默认行为是选一个方向就直接冲。你让它改写一段话，它把整个文章的调性都改了；你让它重组一个文档，它用一种跟你思路完全不搭的方式重新排布。然后你不得不花时间纠正那些你根本没要求改动的地方。

> "在任何重要任务开始前，先给我列出 2-3 种可能的实现方式。等我做出选择后再继续。"

### 指令 3：不确定就说"不确定"

Claude 在承认不确定性之前，会先给你一个自信满满、细节丰富、完全错误的答案。它会用听起来合理的日期、数据、引言来填补空白——感觉都对，实际都不对。

> "如果你对任何事实、数据、日期或引言不确定——在引用之前明确说出来。'我对此不确定'永远比把猜测当作事实呈现要好。永远不要用听起来合理的信息填补空白。"

### 指令 4：回复长度匹配任务复杂度

问一个简单问题，Claude 写四大段；问一个复杂问题，它给一个看起来完整但实质空洞的骨架。两种都没用。

> "回复长度匹配任务复杂度。简单问题给简短的直接答案。复杂任务给完整详细回复。永远不要用重复表述或再次总结你已经说过的话来填充篇幅。"

---

## 四、Part 2：管住 Claude 的行为（指令 5-8）

### 指令 5：大改之前先停住，等确认

你让 Claude 修一个段落，它把整个文档重写了。你让它缩短一段话，它删掉了你实际需要的内容。

> "在对已有内容做出任何重大修改之前——先完全停下。准确描述你准备改什么、为什么改。等我确认后再继续。'我觉得这样更好'不代表你可以直接改。"

### 指令 6：只改我让你改的

Claude 在修一个问题的时候，会顺手"改进"其他五个地方——调整你的措辞、重组你的结构、改写你完全满意的句子。然后你得从头翻一遍，找出它到底动了什么。

> "只改我明确让你改的地方。不要改写、重组或'改进'任何我没有要求改动的内容——即便你认为那样更好。如果你在其他地方发现了值得改进的东西，在末尾提出来。除非我明确要求，不要动它。"

### 指令 7：做完之后告诉我你改了什么

Claude 完成一个任务后，你得盯着输出仔细比对，试图搞清楚跟原来有什么不同——哪些段落被动了？删了什么？加没加你根本没要求的东西？

> "完成任何编辑或写作任务后，最后一定要附一段简短总结：改了什么、没动什么、什么需要我关注。保持简洁——这是状态更新，不是复盘。"

### 指令 8：永远不要替我做决定

随着 AI 工具越来越多地连接到你的邮箱、日历、社交账号、文档，Claude 在你不完全知情的情况下执行一个操作的风险也在同步增长。发一条消息、发布一条内容、安排一项任务——这些行动有真实的后果，而且发生得很快。

> "永远不要代替我发送、发布、分享或安排任何内容，除非我在当前消息中明确确认。'你之前提到过想做这个'不算确认。我必须在当前消息中说'可以'。"

---

## 五、Part 3：告诉 Claude 你是谁（指令 9-11）

### 指令 9：告诉 Claude 你是谁

Claude 不知道你是专家还是新手，是创始人还是自由职业者，想要技术深度还是大白话。不知道，就只能猜。猜对猜错各一半。

> "关于我：姓名、角色、背景、擅长领域、仍在学习的领域。请根据这个背景调整每条回复的深度。永远不要过度解释我已经知道的东西。"

### 指令 10：告诉 Claude 你在做什么

每个会话 Claude 都从零开始——不知道你的项目、不知道目标受众、不知道什么重要。它给你通用输出，是因为它没有其他选择。

> "我正在做的事情：项目、目标、受众、基调、要避免什么。将这些上下文应用到每个任务中。当有内容不符合这个框架时——在继续之前先标出来。"

### 指令 11：锁定你的写作风格

Claude 有一个默认的写作风格。它没问题——但它不是你的。每次用 Claude 写东西，你都得花时间把语言往回掰成你自己的。

> "我的写作风格——始终匹配：语气、句式长度、我惯用的词、我绝不用的词、格式偏好。替我写任何东西时，严格匹配这个风格。不要回退到你自己的默认模式。"

---

## 六、Part 4：让 Claude 记住该记住的（指令 12-15）

这四条是整个清单中最被低估的部分。它们试图解决一个看起来不可能的问题：Claude 在会话之间会忘掉一切——但文件不会。

### 指令 12：让 Claude 维护一个 MEMORY.md 文件

> "维护一个叫 MEMORY.md 的文件。每次做出重要决策后——添加一条记录：决定了什么、为什么、否决了什么替代方案。在每个会话开始时先读取 MEMORY.md。"

### 指令 13：会话结束时的总结——永不丢失进度

你关掉会话，两天后回来，花 15 分钟翻旧消息，试图想起做到哪了、什么做完了、什么还没动。这是完全可以避免的浪费。

> "当我说'session end'或'先到这吧'时——向 MEMORY.md 写一段会话总结：我们做了什么、什么完成了、什么进行中、做了什么决策、下次从哪里继续。"

### 指令 14：记录失败路径——同一个坑不要踩两次

你试了一个 Prompt 方向，花了四次尝试才拿到可用的结果。三周后遇到类似任务，Claude 又从同一个糟糕的建议开始。同样的试错，同样的浪费，每次从零开始。

> "维护一个叫 ERRORS.md 的文件。当一个方法尝试超过两次才成功时——记录什么没做成、什么做成了、下次需要注意什么。在给类似任务建议方案之前先检查 ERRORS.md。"

### 指令 15：给 Claude 一份"永远成立"的事实清单

每个项目都有永久事实——来自过去决策的约束条件、有重要原因存在的规则、不论具体任务是什么都成立的那些事。没有这些，Claude 会随口建议一些跟你的实际情况完全矛盾的东西。

> "以下事实始终成立。应用到每个会话中，无例外：\[你的永久事实\]。如果某个任务与其中任何一条冲突——在继续之前先标出来。"

---

## 七、Part 5：给开发者的六条安全网（指令 16-21）

### 指令 16：严守任务边界

让 Claude 修一个 Bug，它重构了三个文件、重命名了你的变量、重组了你的导入结构、还"改进"了你已经用了几个月的代码。有些改动会出问题，有些会引入过几天才能发现的小差异。

> "只修改跟当前任务直接相关的文件、函数和代码行。不要重构、重命名或'改进'任何我没有明确让你改的东西。如果你在其他地方发现了值得修正的问题——提出来。不要碰。永远不要。"

### 指令 17：不可逆操作前必须确认

Claude Code 会毫不犹豫地删除文件、覆写函数、删库表——因为你让它做的，哪怕你并没有完全意识到自己让它做了。

> "在删除任何文件、覆写已有代码、删除数据库记录或执行任何无法轻松撤销的操作之前——完全停下。准确列出会影响到什么。要求明确确认。只有我在当前消息中说'可以'才继续。"

### 指令 18：硬性阻断——没有例外

部署到生产、在生产数据库上跑迁移、向外部服务发送 API 调用——这些不是"小心点"的场景，是"必须停住等我确认"的场景。

> "以下操作在执行前需要在当前会话明确确认——无例外：部署到任何环境。在任何数据库上运行迁移。发送任何邮件或外部 API 调用。执行任何有不可逆外部副作用的命令。"

### 指令 19：锁定你的技术栈

不定义技术栈，Claude 会建议它见过最多的框架、它默认的包管理器、它认为最流行的库——往往不是你用的、不是你团队熟悉的、跟你已经建好的东西不兼容。

> "技术栈——始终使用这些，除非我主动询问否则不要建议替代方案：语言。框架。包管理器。数据库。测试。代码检查。如果某工具看起来不合适——标出来。但除非我说换，继续用。"

### 指令 20：改完代码后写变更清单

Claude 完成一个编程任务后，你需要弄清楚哪些文件被改了、有没有碰其他东西、有没有留着没做完的。

> "完成任何编程任务后，最后附上：改过的文件——每个文件一行。刻意未触碰文件。需要跟进的事项。保持简洁——这是状态更新，不是复盘。"

### 指令 21：Karpathy 让 CLAUDE.md 爆火的那四条规则

Mayank 在清单最后特别加了一段：Andrej Karpathy——Tesla 前 AI 总监、OpenAI 创始成员——识别出了 Claude Code 在编程任务中最容易出错的四种行为模式。有开发者将它们提炼成了四条指令， **这个文件冲上了 GitHub Trending 第一，将编码准确率从 65% 提升到了 94%** 。

1. **先问，别猜** ——如果有什么不清楚的，在写一行代码之前先问。不要做静默假设
2. **最简方案优先** ——始终实现能跑起来的最简方案。不要加没被要求的抽象
3. **别碰无关代码** ——如果一个文件跟当前任务没有直接关系，不要改它。永远不要
4. **明确说出不确定性** ——如果你对某个方案没信心，在执行前先说明。自信满满的错误比承认不确定造成的危害大得多

---

Mayank 在文末给出了最务实的建议：

> "创建这个文件。粘贴 3 条指令。按需增加。从第一次使用开始，输出质量的差异会马上显现——显而易见。"

指令 1-4 修复沟通方式。指令 5-8 阻止擅改你的内容。指令 9-11 提供上下文，让 Claude 的输出贴合你的实际情况。指令 12-15 给它一个目前最接近真实"记忆"的东西。指令 16-21 让 Claude Code 从不可预测的工具，变成精确可控的工具。
