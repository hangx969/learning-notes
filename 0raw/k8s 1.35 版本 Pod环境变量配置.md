最近在给团队新人做培训的时候，发现大家对Pod环境变量的配置方式理解得比较零散。今天就结合实际项目经验，系统地聊聊这个话题。

## 先说说为什么要用环境变量

去年我们有个项目，开发、测试、生产三套环境的数据库地址都硬编码在代码里，每次发布都要改代码重新打镜像。后来运维老大发飙了："你们这是在玩火！"从那以后，我们老老实实用环境变量了。

## 方式一：直接在Pod里定义

这是最简单直接的方式，适合那些不太会变的配置。

```
apiVersion: v1
kind: Pod
metadata:
  name: myapp-pod
spec:
  containers:
  - name: myapp
    image: myapp:v1.0
    env:
    - name: APP_ENV
      value: "production"
    - name: LOG_LEVEL
      value: "info"
    - name: MAX_CONNECTIONS
      value: "100"
```

这种方式的好处是一目了然，缺点也很明显——每次改个值都要重新apply整个yaml文件。我一般只用它来设置那些"写死"的配置，比如应用名称、版本号之类的。

## 方式二：从ConfigMap读取

ConfigMap是我用得最多的方式，特别适合管理那些经常变动的配置。

先创建一个ConfigMap：

```
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  database.host: "mysql.default.svc.cluster.local"
  database.port: "3306"
  redis.host: "redis.default.svc.cluster.local"
  cache.ttl: "3600"
```

然后在Pod里引用：

```
apiVersion: v1
kind: Pod
metadata:
  name: myapp-pod
spec:
  containers:
  - name: myapp
    image: myapp:v1.0
    env:
    - name: DB_HOST
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: database.host
    - name: DB_PORT
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: database.port
```

如果环境变量比较多，一个个写太麻烦，可以直接把整个ConfigMap导入：

```
apiVersion: v1
kind: Pod
metadata:
  name: myapp-pod
spec:
  containers:
  - name: myapp
    image: myapp:v1.0
    envFrom:
    - configMapRef:
        name: app-config
```

这样ConfigMap里的所有key都会变成环境变量。不过要注意，key的命名要符合环境变量的规范，比如不能有点号，否则会被自动转换成下划线。

## 方式三：从文件加载（EnvFile方式）

这个方式很多人不知道，但在某些场景下特别好用。比如你有一个`.env` 文件：

```
APP_NAME=myapp
APP_VERSION=1.0.0
FEATURE_FLAG_A=true
FEATURE_FLAG_B=false
API_TIMEOUT=30
```

可以先把这个文件创建成ConfigMap：

```
kubectl create configmap env-file-config --from-env-file=.env
```

然后在Pod里这样用：

```
apiVersion: v1
kind: Pod
metadata:
  name: myapp-pod
spec:
  containers:
  - name: myapp
    image: myapp:v1.0
    envFrom:
    - configMapRef:
        name: env-file-config
```

我们团队现在的做法是，开发环境用`.env.dev` ，生产环境用`.env.prod` ，分别创建不同的ConfigMap。部署的时候只需要引用对应的ConfigMap名称就行了。

## 目前做法

结合这几年踩过的坑，目前我们线上环境对于环境变量有一些共识：

**1\. 分层管理**

- 通用配置用ConfigMap
- 敏感信息用Secret（比如数据库密码）
- 固定不变的用直接定义

**2\. 命名规范** 环境变量名统一用大写+下划线，比如 `DB_HOST` 而不是 `dbHost` 。这样在代码里一眼就能看出来是环境变量。

**3\. 设置默认值** 在应用代码里一定要给环境变量设置默认值，避免因为配置缺失导致应用启动失败。

**4\. 文档化** 我们现在的做法是在项目根目录放一个 `env.example` 文件，列出所有需要的环境变量和说明。新人来了看这个文件就知道要配置什么了。

最后给个我们实际在用的配置，综合了上面几种方式：

```
apiVersion: v1
kind: ConfigMap
metadata:
  name: myapp-config
data:
  APP_ENV: "production"
  LOG_LEVEL: "info"
  DB_HOST: "mysql.default.svc.cluster.local"
  REDIS_HOST: "redis.default.svc.cluster.local"
---
apiVersion: v1
kind: Pod
metadata:
  name: myapp-pod
spec:
  containers:
  - name: myapp
    image: myapp:v1.0
    env:
    - name: POD_NAME
      valueFrom:
        fieldRef:
          fieldPath: metadata.name
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-secret
          key: password
    envFrom:
    - configMapRef:
        name: myapp-config
```

这个配置里，固定的配置放ConfigMap，敏感信息用Secret，Pod自身的信息用fieldRef。这样既安全又灵活。

## 方式4（新方式）：1.35 EnvFiles

使用 ConfigMap 和 Secret 来设置环境变量，这引入了额外的复杂性。例如，您需要分别管理工作负载的 Pod 及其配置，同时确保配置和工作负载 Pod 的有序更新。

或者，您可能正在使用供应商提供的容器，该容器需要环境变量（例如许可证密钥或一次性令牌），但您不想为了完成工作而硬编码这些变量或挂载卷。

k8s 1.35 提供了一种叫做 EnvFiles的新功能， 启用此功能后，您可以指示 kubelet 加载容器的资源。 从卷中获取环境变量（该卷必须是 Pod 的一部分）。 此功能门允许您直接从 emptyDir 卷中的文件加载环境变量。 无需将该文件实际挂载到容器中。

此功能的核心在于允许您将容器指向一个由 initContainer `生成的文件，并让 Kubernetes 解析该文件以设置环境变量。该文件位于` `emptyDir` 卷（一个与 Pod 生命周期相同的临时存储空间）中。您的主容器无需挂载该卷。kubelet 将在容器启动时读取该文件并注入这些变量。

举个简单的例子：

```
apiVersion: v1
kind: Pod
spec:
  initContainers:
  - name: generate-config
    image: busybox
    command: ['sh', '-c', 'echo "CONFIG_VAR=HELLO" > /config/config.env']
    volumeMounts:
    - name: config-volume
      mountPath: /config
  containers:
  - name: app-container
    image: gcr.io/distroless/static
    env:
    - name: CONFIG_VAR
      valueFrom:
        fileKeyRef:
          path: config.env
          volumeName: config-volume
          key: CONFIG_VAR
  volumes:
  - name: config-volume
    emptyDir: {}
```

使用这种方法非常简单。您只需在 Pod 规范中使用 `fileKeyRef` 字段定义环境变量，该字段会告诉 Kubernetes 在哪里可以找到文件以及要提取哪个键。

**目前如果新部署 k8s 1.35 版本的集群，建议大家都采用新的方式，再也不用管理一堆 configMap 啦！**
