# 高可用架构

## 定义

高可用架构（High Availability, HA）是通过冗余设计、故障检测和自动故障转移等机制，消除系统中的单点故障（SPOF），使服务在部分组件失效时仍能持续对外提供服务的架构模式。其核心目标是最大化系统的可用时间（uptime），通常以"几个 9"的可用性指标来衡量（如 99.99% 即年停机不超过 52 分钟）。

## 核心要点

- **消除单点故障**：任何关键组件（控制面、数据库、负载均衡器）都必须有冗余副本，单个节点故障不影响整体服务
- **多 Master 控制面**：K8s 生产环境最低 3 Master 节点，API Server、Scheduler、Controller Manager 均多副本运行
- **etcd 集群化**：etcd 采用 Raft 共识算法，需奇数节点（通常 3 或 5 个），容忍 (n-1)/2 个节点故障；大规模集群（250+ 节点）建议 etcd 与控制面分离部署
- **负载均衡与 VIP**：通过 keepalived 提供虚拟 IP（VIP）+ Nginx/HAProxy 实现 API Server 的负载均衡，客户端通过 VIP 透明访问
- **数据库主从复制**：MySQL 主从复制实现读写分离和数据冗余；MySQL NDB Cluster 通过管理节点、数据节点、SQL 节点三角色实现分布式高可用
- **Redis 高可用三模式**：主从复制（数据冗余）、Sentinel 哨兵（自动故障转移）、Cluster 集群（16384 哈希槽数据分片 + 3 主 3 从），K8s 环境推荐 Cluster 模式
- **PostgreSQL HA**：Pgpool（流量接入与读写分离）+ repmgr（主从自动切换）+ witness（偶数节点防脑裂）
- **MongoDB 副本集**：通过副本集（Replica Set）实现自动故障转移，分片（Sharding）实现水平扩展
- **跨地域高可用**：从单集群多 Master 到两地三中心（主备模式）再到异地多活（多中心同时承载流量），架构复杂度和可用性逐步提升

## 与其他概念的关系

- [[KnowledgeBase/entities/Kubernetes|Kubernetes]]：多 Master 高可用控制面是 K8s 生产部署的基本要求，3 Master 可管理约 900 个 Worker 节点
- [[KnowledgeBase/entities/Redis|Redis]]：Sentinel 提供监控和自动故障转移，Cluster 模式通过数据分片实现高可用和水平扩展
- [[KnowledgeBase/entities/MySQL|MySQL]]：主从复制、NDB Cluster、KubeBlocks RaftGroup 等多种 HA 方案
- [[KnowledgeBase/entities/PostgreSQL|PostgreSQL]]：Pgpool + repmgr + witness 组成完整的 HA 方案
- [[KnowledgeBase/entities/Harbor|Harbor]]：原生不集成高可用，需外部 HA 的 Redis 和 PostgreSQL 支撑
- [[KnowledgeBase/entities/Kafka|Kafka]]：Strimzi Operator 管理 Kafka 集群，Mirror Maker 实现跨集群数据复制和灾备
- [[KnowledgeBase/entities/Rancher|Rancher]]：企业级多集群管理平台，本身支持高可用部署
